# Architecture — FreightConnect

> **Target architecture** — this repo is greenfield; the code does not exist yet.
> This file describes how it should be built. Claude Code reads this before touching architectural code.

## 1. System overview

FreightConnect is a single Next.js (App Router) application that serves both the browser UI and the JSON API (via Route Handlers under `app/api/`). It is SSR-first because the directory and profile pages are the organic-discovery surface (SEO is the growth engine). Data lives in PostgreSQL via Prisma. KYC documents live in a private S3-compatible bucket and are reachable only through short-lived signed URLs generated by admin-only code. Transactional email (verification, inquiry notifications, approval/rejection) goes through a provider behind a thin interface so it can be swapped.

There is **one surface: the web app.** There is no standalone backend service and no browser extension. "API" here means Next.js Route Handlers in the same deployable.

## 3. Data model

Primary entities and their relationships (full spec in @docs/PRD.md §5):

```
User (role: shipper | forwarder | admin)
  └─1:1─ ForwarderProfile (status: draft|pending|approved|rejected|suspended, verified, slug)
            ├─1:N─ CountryCoverage (country, city, isHeadquarters, ports[])
            ├─1:N─ TradeLane (originCountry, destinationCountry)
            ├─1:N─ KycDocument (type, storageKey, mimeType, sizeBytes)   ← PRIVATE
            ├─1:N─ Inquiry (shipper contact + lane + mode + status)       ← shipper may be anonymous
            └─1:N─ Review (P2)
Taxonomy: Country, Port, Mode, Service (reference/lookup)
```

Key constraints / invariants:
- A profile is publicly visible **only** when `status = approved`.
- `verified = true` ⇒ `status = approved` (never verified while pending/rejected).
- `slug` is unique and immutable once approved; generated server-side.
- KYC documents are never exposed on any public endpoint.
- An `Inquiry` is persisted before any email send is attempted.
- `modes` / `services` reference Taxonomy codes; index `status`, `country`, `mode`, `service`, and lane columns.

Migrations live in `prisma/migrations/`. **Never edit applied migrations** — add a new one.

## 4. Service boundaries

Logical modules inside the single app (not separate deployables):

| Module | Responsibility | Depends on |
|--------|----------------|------------|
| `lib/auth` | Session, roles, email verification, password hashing | Prisma, email |
| `lib/forwarders` | Profile CRUD, slug generation, status transitions, public projection | Prisma, storage |
| `lib/inquiries` | Create + persist inquiry, queue notification, idempotency, rate limit | Prisma, email |
| `lib/admin` | Review queue, approve/reject/suspend, signed-URL issuance, audit log | Prisma, storage |
| `lib/storage` | Private bucket put/get, signed URL generation | S3/R2 |
| `lib/email` | Provider-agnostic send interface + retry queue | provider |
| `lib/validation` | Zod schemas at every boundary (filters, bodies, uploads) | — |

**Trust boundary:** Route Handlers validate input (Zod) and derive identity from the session before calling any `lib/*` service. Services never receive a client-supplied owner/tenant id.

## 5. External integrations

| Integration | Purpose | Auth | Failure mode |
|-------------|---------|------|--------------|
| Transactional email (Resend/SES/Postmark) | Inquiry/verification/approval emails | server-side API key | Persist first; queue + retry; never lose a lead |
| Object storage (S3/R2) | Private KYC doc storage + signed URLs | IAM/key | Fail upload 5xx; no orphaned DB record without object |
| Captcha (Turnstile/hCaptcha) | Inquiry/registration spam control | site+secret key | Fail closed on inquiry submit |

Providers are TODO at the interface level — `lib/email` and `lib/storage` abstract them.

## 6. Key decisions (ADRs)

### ADR-001: One Next.js app, not a separate API service
- **Context:** Directory + profile pages must be SSR for SEO; the same team owns UI and API.
- **Decision:** Serve API from Next.js Route Handlers in the same deployable.
- **Consequences:** Simpler deploy + shared types; "API surface" is verified with the browser-verifier and integration tests, not a separate service harness.

### ADR-002: Postgres full-text + indexed filters for v1 search
- **Context:** Filter-first directory at ≤10k profiles.
- **Decision:** Use Postgres indexed filter columns + full-text for `q`; defer Meilisearch/Typesense.
- **Consequences:** No extra search infra in v1; revisit if free-text relevance becomes a need.

### ADR-003: Anonymous shipper inquiries allowed in v1
- **Context:** Low-friction lead capture (PRD open question, current ASSUMPTION).
- **Decision:** Shippers may submit an inquiry with email only; no User row required.
- **Consequences:** Inquiry stores contact details directly; revisit if accounts become required.

## 7. Error handling philosophy

- API errors return a consistent envelope: `{ error: "<CODE>", field?: "<name>", message?: "<human>" }` with the documented HTTP status (e.g. `400 INVALID_FILTER`, `403 FORBIDDEN`, `404`, `409 ALREADY_APPROVED`, `413`, `429 RATE_LIMITED`).
- Logging is centralized in `lib/logger`; tokens/secrets are redacted.
- User-facing surfaces show friendly messages + retry affordances; never a stack trace.
- Errors are never silently swallowed — log with context, surface appropriately.

## 8. Performance & scale assumptions

- Directory search (`/api/forwarders`) p95 < 500 ms at up to 10k profiles (requires indexed filter columns).
- Profile page SSR p95 < 800 ms.
- Paginate every list endpoint (default 20, cap 200). Debounce free-text search.

## 9. Security notes

- **Secrets:** env / Vercel project env. Never in client bundle; never `NEXT_PUBLIC_` for secrets.
- **Auth:** session cookies `httpOnly`, `secure`, `sameSite=lax`; roles enforced server-side on every protected route and handler; passwords hashed (argon2/bcrypt); no user enumeration.
- **PII:** KYC docs are PII-adjacent → private bucket, signed URLs, access-logged. Shipper inquiry contact details are PII → never publicly listed.
- **Audit:** every admin approve/reject/suspend logged with actor + timestamp.
- **GDPR-aware:** support data export + deletion for accounts; document KYC/inquiry retention (jurisdictions TBD — see PRD open questions).
